Shardwire is a Discord-first bridge: one **Node process** holds your bot and Discord connection, and another **Node process** (or several) connects over a WebSocket to react to events and call typed actions—without sharing the same memory as `discord.js` client code.

If you are new here, you do not need the whole architecture first.

Follow the [**Tutorial**](/docs/tutorial/lesson-0-overview) for a **guided path** (two terminals, env vars, first `!ping`). It is the fastest way to see Shardwire working end to end.

## What lives where

- The **bot process** runs `createBotBridge()` and owns Discord connectivity, intents, event normalization, and action execution.
- The **app process** runs `connectBotBridge()` and consumes typed events while invoking typed actions across the bridge.
- **Scoped secrets** and **negotiated capabilities** define what each app process is allowed to see and do.

## Why teams reach for it

- Split-process architecture without inventing a custom event bus.
- Capability-aware actions and subscriptions instead of ad-hoc permission checks.
- Strict startup that can reject mismatched intents, secret scope, and subscriptions before traffic hits your app.
- A public API surface small enough to reason about but broad enough to support real apps.
