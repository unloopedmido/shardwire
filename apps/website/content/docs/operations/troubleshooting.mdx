title: "Troubleshooting"
description: "Work failures from connection and capability negotiation outward; most issues become obvious once you inspect the negotiated surface."
---

## If the app cannot connect

- verify the URL ends with `/shardwire` or your configured custom path
- verify the secret matches a configured bridge secret
- verify `ws://` is only being used on loopback

## If handlers never fire

- inspect `app.capabilities().events`
- verify the required intents are enabled on `createBotBridge()`
- confirm the scoped secret allows that event
- confirm the subscription event name is correct

## If an action fails

- branch on `result.ok`
- log `result.error.code`
- inspect `result.error.details`
- confirm the action is included in negotiated capabilities

## If strict startup fails

- read the attached diagnosis report
- compare manifest events to bot intents
- compare manifest actions to scoped secret permissions
- compare runtime subscriptions to declared manifest filters

## Vite and zlib-sync

Bundlers pre-bundle the main `shardwire` entry, which includes the bot runtime (`discord.js`). `discord.js` references the optional native module `zlib-sync`, which Vite cannot resolve for the browser (error text: `Failed to resolve import "zlib-sync"`).

**Fix:** import app-side APIs from the dedicated client entry instead of the package root:

```ts
import { connectBotBridge, defineShardwireApp } from 'shardwire/client';
```

Keep `createBotBridge` imports on `shardwire` in **Node-only** bot processes. `@shardwire/react` resolves `shardwire/client` via its **`shardwire`** dependency; use **`shardwire` ≥ 2.0.0** on **Node.js 22+** when you need current browser/Vite fixes (`zlib-sync`, **`WebSocket`**, **`crypto`** / request IDs).

## Vite and `crypto` / `node:crypto`

Older client bundles imported **`node:crypto`** for UUIDs used by **`app.actions.*`**, which makes Vite warn or treat **`crypto`** as external and can break actions in the browser.

**Fix:** upgrade to **`shardwire` ≥ 2.0.0**, which uses **`globalThis.crypto`** only and declares **Node.js 22+** in **`engines`**. Upgrade Node if you still see missing **`globalThis.crypto`** errors.

## Browser or Vite: `WebSocket is not a constructor`

The app bridge uses the **standard `globalThis.WebSocket` constructor** (same as browsers and Node.js 22+). It does **not** use the npm `ws` package in the browser; bundling `ws` for the client often yields a non-constructor shim and errors like `import_browser.WebSocket is not a constructor`.

**Fix:** use **`shardwire` ≥ 2.0.0** so the client build omits `ws` for `connectBotBridge` and matches supported **Node.js 22+**. No app code changes are required beyond upgrading Node and the package.

## Node.js below 22: `connectBotBridge requires globalThis.WebSocket`

**Shardwire requires Node.js 22+** for app processes: **`globalThis.WebSocket`** is part of the supported runtime contract (same as modern browsers). Upgrade Node to **22 or newer**; polyfilling **`WebSocket`** on older Node versions is not supported.

Treat troubleshooting as a capability audit first, not a transport guess.
